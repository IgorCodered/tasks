# Техническое задание: TODO Manager

## 1. Общие сведения
**TODO Manager** — это веб-приложение для управления личными задачами, позволяющее пользователям создавать, классифицировать и отслеживать выполнение дел.

## 2. Бизнес-правила
1. **Авторизация:** Доступ к задачам осуществляется только после регистрации и входа в систему.
2. **Изоляция данных:** Пользователь видит только свои задачи.
3. **Статус задачи:** Задача может находиться в одном из трех состояний: `Новая`, `В процессе`, `Выполнена`.
4. **Приоритеты:** Каждая задача имеет уровень приоритета: `Низкий`, `Средний`, `Высокий`.
5. **Удаление:** Удаление задачи является безвозвратным.

## 3. Функциональные требования (ФТ)
*   **ФТ-1:** Регистрация и аутентификация пользователей (JWT).
*   **ФТ-2:** Создание задачи (название, описание, дедлайн, приоритет).
*   **ФТ-3:** Просмотр списка задач с возможностью фильтрации по статусу и приоритету.
*   **ФТ-4:** Редактирование и изменение статуса задачи.
*   **ФТ-5:** Удаление задачи.
*   **ФТ-6:** Поиск задач по ключевым словам в названии.

## 4. Нефункциональные требования (НФТ)
*   **НФТ-1 (Производительность):** Время отклика API не должно превышать 200 мс при стандартной нагрузке.
*   **НФТ-2 (Безопасность):** Пароли пользователей должны храниться в зашифрованном виде (bcrypt).
*   **НФТ-3 (Масштабируемость):** Архитектура должна поддерживать до 10,000 активных пользователей.
*   **НФТ-4 (Доступность):** Приложение должно быть доступно 99.9% времени (Uptime).
*   **НФТ-5 (Уведомления):** Система должна гарантировать доставку критических напоминаний в течение 30 секунд.
*   **НФТ-6 (Персонализация):** Алгоритм машинного обучения должен адаптироваться к паттернам пользователя за 14 дней использования.
*   **НФТ-7 (Надежность):** Система должна иметь fallback-механизмы при недоступности основных каналов уведомлений.

## 5. Аналитика и метрики

### 5.1. Фича: Трекинг Daily Active Users (DAU)

**Входные данные:**
- `user_id` (UUID) — идентификатор пользователя
- `session_id` (UUID) — идентификатор сессии
- `timestamp` (DateTime) — время активности
- `action_type` (String) — тип действия (login, view_task, create_task, complete_task, etc.)
- `device_info` (JSON) — информация об устройстве

**Выходные данные:**
- `date` (Date) — дата отчётности
- `dau_count` (Integer) — количество уникальных пользователей
- `dau_by_action` (JSON) — разбивка по типам действий
- `dau_trend` (Array) — динамика за последние 7 дней

**Бизнес-логика:**
1. Пользователь считается активным, если выполнил хотя бы одно действие в течение календарного дня (00:00–23:59 по UTC).
2. Повторные действия одного пользователя в течение дня учитываются один раз.
3. Действия анонимных пользователей не учитываются.
4. Расчёт производится ежедневно в 00:05 UTC за предыдущий день.

**Примеры JSON:**

*Входные данные (POST /api/analytics/events)*
```json
{
  "user_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "session_id": "sess_9876543210abcdef",
  "timestamp": "2024-01-15T14:32:18Z",
  "action_type": "complete_task",
  "device_info": {
    "platform": "web",
    "browser": "Chrome",
    "os": "Windows"
  }
}
```

*Выходные данные (GET /api/analytics/dau?date=2024-01-15)*
```json
{
  "date": "2024-01-15",
  "dau_count": 1247,
  "dau_by_action": {
    "login": 892,
    "view_task": 1105,
    "create_task": 456,
    "complete_task": 378,
    "delete_task": 89
  },
  "dau_trend": [
    {"date": "2024-01-09", "count": 1189},
    {"date": "2024-01-10", "count": 1234},
    {"date": "2024-01-11", "count": 1156},
    {"date": "2024-01-12", "count": 1298},
    {"date": "2024-01-13", "count": 987},
    {"date": "2024-01-14", "count": 1045},
    {"date": "2024-01-15", "count": 1247}
  ]
}
```

**Поэтапная разработка:**
- **Этап 1:** Сбор событий активности пользователей (логирование действий)
- **Этап 2:** Агрегация данных по дням и расчёт DAU
- **Этап 3:** Построение трендов и визуализация динамики
- **Этап 4:** Оптимизация производительности для больших объёмов данных

### 5.2. Фича: Task Completion Rate

**Входные данные:**
- `task_id` (UUID) — идентификатор задачи
- `user_id` (UUID) — идентификатор пользователя
- `initial_status` (Enum) — начальный статус задачи
- `final_status` (Enum) — конечный статус задачи
- `status_changed_at` (DateTime) — время изменения статуса
- `created_at` (DateTime) — время создания задачи

**Выходные данные:**
- `period` (String) — период отчёта (last_7_days, last_30_days, all_time)
- `total_tasks_created` (Integer) — всего создано задач
- `total_tasks_completed` (Integer) — всего завершено задач
- `completion_rate` (Decimal) — процент завершения (0–1)
- `completion_rate_by_priority` (JSON) — разбивка по приоритетам
- `avg_completion_time_hours` (Decimal) — среднее время выполнения в часах

**Бизнес-логика:**
1. Задача считается завершённой, если её статус изменён на «Выполнена».
2. В расчёт включаются только задачи, созданные в пределах отчётного периода.
3. Если задача была создана и завершена в одном периоде, она учитывается в числителе и знаменателе.
4. Задачи со статусом «Новая» или «В процессе» не учитываются в числителе.
5. Расчёт производится по формуле: `(кол-во завершённых задач) / (кол-во созданных задач)`.

**Примеры JSON:**

*Входные данные (внутреннее событие изменения статуса)*
```json
{
  "task_id": "task_abc123xyz789",
  "user_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "initial_status": "В процессе",
  "final_status": "Выполнена",
  "status_changed_at": "2024-01-15T16:45:00Z",
  "created_at": "2024-01-14T10:00:00Z"
}
```

*Выходные данные (GET /api/analytics/task-completion?period=last_30_days)*
```json
{
  "period": "last_30_days",
  "total_tasks_created": 542,
  "total_tasks_completed": 387,
  "completion_rate": 0.71,
  "completion_rate_by_priority": {
    "Низкий": 0.65,
    "Средний": 0.72,
    "Высокий": 0.84
  },
  "avg_completion_time_hours": 18.5
}
```

**Поэтапная разработка:**
- **Этап 1:** Логирование изменений статуса задач
- **Этап 2:** Расчёт completion rate за фиксированный период
- **Этап 3:** Сегментация по приоритетам и другим параметрам
- **Этап 4:** Расчёт среднего времени выполнения задач

### 5.3. Фича: Retention Rate

**Входные данные:**
- `user_id` (UUID) — идентификатор пользователя
- `registration_date` (Date) — дата регистрации
- `activity_dates` (Array of Date) — даты активности пользователя
- `cohort` (String) — когорта пользователя (неделя/месяц регистрации)

**Выходные данные:**
- `cohort_period` (String) — период когорты (например, 2024-W02)
- `registered_users` (Integer) — количество зарегистрированных пользователей в когорте
- `retention_day_1` (Decimal) — процент вернувшихся через 1 день
- `retention_day_7` (Decimal) — процент вернувшихся через 7 дней
- `retention_day_30` (Decimal) — процент вернувшихся через 30 дней
- `retention_curve` (Array) — кривая удержания по дням

**Бизнес-логика:**
1. Пользователь считается вернувшимся, если он проявил активность в целевой день (см. логику DAU).
2. Когорта формируется по неделе регистрации пользователя.
3. Retention Rate рассчитывается как `(кол-во вернувшихся пользователей) / (кол-во пользователей в когорте)`.
4. День 0 — это день регистрации.
5. Расчёт производится еженедельно для всех активных когорт.

**Примеры JSON:**

*Входные данные (агрегированные данные по пользователю)*
```json
{
  "user_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "registration_date": "2024-01-08",
  "activity_dates": ["2024-01-08", "2024-01-09", "2024-01-15", "2024-01-22"],
  "cohort": "2024-W02"
}
```

*Выходные данные (GET /api/analytics/retention?cohort=2024-W02)*
```json
{
  "cohort_period": "2024-W02",
  "registered_users": 342,
  "retention_day_1": 0.68,
  "retention_day_7": 0.45,
  "retention_day_30": 0.28,
  "retention_curve": [
    {"day": 0, "rate": 1.0},
    {"day": 1, "rate": 0.68},
    {"day": 2, "rate": 0.59},
    {"day": 3, "rate": 0.54},
    {"day": 4, "rate": 0.51},
    {"day": 5, "rate": 0.48},
    {"day": 6, "rate": 0.46},
    {"day": 7, "rate": 0.45}
  ]
}
```

**Поэтапная разработка:**
- **Этап 1:** Формирование когорт по дате регистрации
- **Этап 2:** Отслеживание активности пользователей по дням
- **Этап 3:** Расчёт retention для ключевых точек (D1, D7, D30)
- **Этап 4:** Построение полной кривой удержания

### 5.4. Фича: Notification Effectiveness Rate

**Входные данные:**
- `reminder_id` (UUID) — идентификатор напоминания
- `task_id` (UUID) — идентификатор задачи
- `user_id` (UUID) — идентификатор пользователя
- `notification_sent_at` (DateTime) — время отправки уведомления
- `notification_delivered_at` (DateTime) — время доставки
- `task_completed_at` (DateTime) — время выполнения задачи (если применимо)
- `channel` (Enum) — канал уведомления (email, push, sms)

**Выходные данные:**
- `period` (String) — период отчёта
- `total_notifications_sent` (Integer) — всего отправлено уведомлений
- `tasks_completed_after_notification` (Integer) — задач выполнено после уведомления
- `effectiveness_rate` (Decimal) — процент эффективности (0–1)
- `effectiveness_by_channel` (JSON) — эффективность по каналам
- `avg_time_to_complete_minutes` (Decimal) — среднее время до выполнения

**Бизнес-логика:**
1. Уведомление считается эффективным, если задача была выполнена в течение 24 часов после доставки уведомления.
2. Если по задаче было отправлено несколько уведомлений, учитывается первое, после которого задача была выполнена.
3. Задачи, выполненные до отправки уведомления, не учитываются.
4. Расчёт по формуле: `(задач выполнено после уведомления) / (всего уведомлений доставлено)`.

**Примеры JSON:**

*Входные данные (событие доставки уведомления и выполнения задачи)*
```json
{
  "reminder_id": "rem_xyz789abc123",
  "task_id": "task_abc123xyz789",
  "user_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "notification_sent_at": "2024-01-15T08:00:00Z",
  "notification_delivered_at": "2024-01-15T08:00:15Z",
  "task_completed_at": "2024-01-15T09:30:00Z",
  "channel": "push"
}
```

*Выходные данные (GET /api/analytics/notification-effectiveness?period=last_30_days)*
```json
{
  "period": "last_30_days",
  "total_notifications_sent": 2847,
  "tasks_completed_after_notification": 2195,
  "effectiveness_rate": 0.77,
  "effectiveness_by_channel": {
    "email": 0.65,
    "push": 0.82,
    "sms": 0.91
  },
  "avg_time_to_complete_minutes": 47
}
```

**Поэтапная разработка:**
- **Этап 1:** Логирование отправки и доставки уведомлений
- **Этап 2:** Корреляция уведомлений с выполнением задач
- **Этап 3:** Расчёт эффективности по каждому каналу
- **Этап 4:** Анализ времени реакции на уведомления

### 5.5. Фича: Average Response Time

**Входные данные:**
- `reminder_id` (UUID) — идентификатор напоминания
- `notification_delivered_at` (DateTime) — время доставки уведомления
- `user_action_at` (DateTime) — время действия пользователя (открытие, клик, выполнение задачи)
- `action_type` (Enum) — тип действия (open, click, complete_task, snooze)

**Выходные данные:**
- `period` (String) — период отчёта
- `avg_response_time_minutes` (Decimal) — среднее время отклика в минутах
- `median_response_time_minutes` (Decimal) — медианное время отклика
- `response_time_by_channel` (JSON) — время отклика по каналам
- `response_time_distribution` (JSON) — распределение по временным интервалам

**Бизнес-логика:**
1. Время отклика измеряется как разница между доставкой уведомления и первым действием пользователя.
2. Если пользователь не совершил действие в течение 24 часов, уведомление исключается из расчёта.
3. Действия snooze (отложить) также учитываются как отклик.
4. Расчёт производится отдельно для каждого канала уведомлений.

**Примеры JSON:**

*Входные данные (событие действия пользователя)*
```json
{
  "reminder_id": "rem_xyz789abc123",
  "notification_delivered_at": "2024-01-15T08:00:15Z",
  "user_action_at": "2024-01-15T08:47:30Z",
  "action_type": "complete_task"
}
```

*Выходные данные (GET /api/analytics/response-time?period=last_30_days)*
```json
{
  "period": "last_30_days",
  "avg_response_time_minutes": 45.3,
  "median_response_time_minutes": 32.0,
  "response_time_by_channel": {
    "email": 78.5,
    "push": 23.1,
    "sms": 12.4
  },
  "response_time_distribution": {
    "0-15_min": 0.32,
    "15-30_min": 0.28,
    "30-60_min": 0.22,
    "60-120_min": 0.12,
    "120+_min": 0.06
  }
}
```

**Поэтапная разработка:**
- **Этап 1:** Фиксация времени доставки уведомлений
- **Этап 2:** Отслеживание действий пользователей после уведомлений
- **Этап 3:** Расчёт среднего и медианного времени отклика
- **Этап 4:** Построение распределения по временным интервалам

### 5.6. Фича: Snooze Rate

**Входные данные:**
- `reminder_id` (UUID) — идентификатор напоминания
- `task_id` (UUID) — идентификатор задачи
- `user_id` (UUID) — идентификатор пользователя
- `original_trigger_time` (DateTime) — оригинальное время срабатывания
- `snooze_until` (DateTime) — время повторного срабатывания
- `snooze_count` (Integer) — количество откладываний данной задачи

**Выходные данные:**
- `period` (String) — период отчёта
- `total_reminders_triggered` (Integer) — всего сработавших напоминаний
- `total_snoozed_reminders` (Integer) — всего отложенных напоминаний
- `snooze_rate` (Decimal) — процент откладываний (0–1)
- `avg_snooze_count_per_task` (Decimal) — среднее кол-во откладываний на задачу
- `snooze_by_time_of_day` (JSON) — частота откладываний по времени суток

**Бизнес-логика:**
1. Напоминание считается отложенным, если пользователь активировал функцию snooze.
2. Максимальное количество откладываний одной задачи ограничено (настраиваемый параметр, по умолчанию 3).
3. Snooze Rate рассчитывается как `(отложенных напоминаний) / (всего сработавших напоминаний)`.
4. Интервал откладывания может быть фиксированным (5, 15, 30 минут) или адаптивным.


*Выходные данные (GET /api/analytics/snooze-rate?period=last_30_days)*
```json
{

  "total_reminders_triggered": 2847,
  "total_snoozed_reminders": 627,
  "snooze_rate": 0.22,
  "avg_snooze_count_per_task": 1.4,
  "snooze_by_time_of_day": {
    "06:00-09:00": 0.15,
    "09:00-12:00": 0.18,
    "12:00-15:00": 0.24,
    "15:00-18:00": 0.28,
    "18:00-21:00": 0.25,
    "21:00-24:00": 0.19
  }
}
```

**Поэтапная разработка:**
- **Этап 1:** Реализация функции snooze с базовыми интервалами
- **Этап 2:** Логирование всех случаев откладывания
- **Этап 3:** Расчёт snooze rate и статистики по задачам
- **Этап 4:** Анализ паттернов откладывания по времени суток

### 5.7. Фича: Channel Performance

**Входные данные:**
- `reminder_id` (UUID) — идентификатор напоминания
- `channel` (Enum) — канал уведомления (email, push, sms)
- `sent_at` (DateTime) — время отправки
- `delivered_at` (DateTime) — время доставки (если применимо)
- `opened_at` (DateTime) — время открытия (для email)
- `clicked_at` (DateTime) — время клика
- `task_completed_at` (DateTime) — время выполнения задачи

**Выходные данные:**
- `period` (String) — период отчёта
- `channels` (Array) — массив данных по каналам
  - `channel_name` (String) — название канала
  - `sent_count` (Integer) — отправлено
  - `delivered_count` (Integer) — доставлено
  - `delivery_rate` (Decimal) — процент доставки
  - `open_rate` (Decimal) — процент открытий (для email)
  - `click_rate` (Decimal) — процент кликов
  - `conversion_rate` (Decimal) — процент выполнения задач

**Бизнес-логика:**
1. Delivery Rate рассчитывается только для каналов с подтверждением доставки (push, sms).
2. Open Rate применим только для email-уведомлений.
3. Conversion Rate — процент задач, выполненных в течение 24 часов после клика/открытия.
4. Каналы ранжируются по conversion rate для формирования рекомендаций.

**Примеры JSON:**

*Входные данные (лог уведомления)*
```json
{
  "reminder_id": "rem_xyz789abc123",
  "channel": "email",
  "sent_at": "2024-01-15T08:00:00Z",
```

### 5.8. Фича: Optimal Timing Score

**Входные данные:**
- `user_id` (UUID) — идентификатор пользователя
- `reminder_id` (UUID) — идентификатор напоминания
- `scheduled_time` (DateTime) — запланированное время напоминания
- `actual_completion_time` (DateTime) — фактическое время выполнения
- `hour_of_day` (Integer) — час суток (0–23)
- `day_of_week` (Integer) — день недели (1–7)
- `is_effective` (Boolean) — было ли напоминание эффективным

**Выходные данные:**
- `user_id` (UUID) — идентификатор пользователя
- `optimal_time_slots` (Array) — рекомендованные временные слоты
  - `time_range` (String) — диапазон времени (например, "08:00-10:00")
  - `score` (Decimal) — оценка эффективности (0–10)
  - `sample_size` (Integer) — количество наблюдений
- `best_day_of_week` (Integer) — лучший день недели

**Бизнес-логика:**
1. Система анализирует исторические данные о выполнении задач после напоминаний.
2. Временной слот считается эффективным, если tasks completed within 2 hours >= 70%.
3. Минимальный размер выборки для рекомендации — 10 наблюдений на слот.
4. Оценка рассчитывается как взвешенная средняя effectiveness rate и response time.
5. Рекомендации обновляются еженедельно на основе новых данных.

## 8. Умная система напоминаний и уведомлений

### 8.1. Фича: Создание персонализированных напоминаний

**Входные данные:**
- `task_id` (UUID) — идентификатор задачи
- `user_id` (UUID) — идентификатор пользователя
- `reminder_type` (Enum) — тип напоминания: `static`, `relative`, `smart`, `escalation`
- `trigger_time` (DateTime) — конкретное время срабатывания (для static)
- `offset_minutes` (Integer) — смещение в минутах от дедлайна (для relative)
- `channels` (Array) — каналы доставки: `email`, `push`, `sms`
- `is_recurring` (Boolean) — является ли повторяющимся
- `recurrence_pattern` (Enum) — паттерн повторения: `daily`, `weekly`, `monthly`
- `snooze_enabled` (Boolean) — разрешена ли функция snooze
- `max_attempts` (Integer) — максимальное количество попыток

**Выходные данные:**
- `reminder_id` (UUID) — идентификатор созданного напоминания
- `task_id` (UUID) — идентификатор задачи
- `reminder_type` (Enum) — тип напоминания
- `next_trigger` (DateTime) — время следующего срабатывания
- `status` (Enum) — статус: `active`, `snoozed`, `completed`, `cancelled`
- `channels` (Array) — выбранные каналы доставки
- `created_at` (DateTime) — дата создания

**Бизнес-логика:**
1. Напоминание может быть создано только для существующей задачи.
2. Для типа `relative` время срабатывания вычисляется автоматически на основе дедлайна задачи.
3. Для типа `smart` система предлагает оптимальное время на основе истории пользователя.
4. Повторяющиеся напоминания создаются с указанным интервалом до выполнения задачи.
5. Максимальное количество попыток ограничивает число отправлений для одной задачи.
6. Пользователь не может создать напоминание в прошлом.

**Примеры JSON:**

*Входные данные (POST /api/reminders)*
```json
{
  "task_id": "task_abc123xyz789",
  "user_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "reminder_type": "relative",
  "offset_minutes": 60,
  "channels": ["email", "push"],
  "is_recurring": false,
  "snooze_enabled": true,
  "max_attempts": 3
}
```

*Выходные данные (ответ на создание напоминания)*
```json
{
  "reminder_id": "rem_xyz789abc123",
  "task_id": "task_abc123xyz789",
  "reminder_type": "relative",
  "next_trigger": "2024-01-15T08:00:00Z",
  "status": "active",
  "channels": ["email", "push"],
  "created_at": "2024-01-14T10:00:00Z"
}
```

**Поэтапная разработка:**
- **Этап 1:** CRUD операции для напоминаний
- **Этап 2:** Поддержка типов static и relative
- **Этап 3:** Реализация recurring напоминаний
- **Этап 4:** Интеграция с каналами доставки

### 8.2. Фича: Автоматические напоминания о дедлайнах

**Входные данные:**
- `task_id` (UUID) — идентификатор задачи
- `due_date` (DateTime) — дедлайн задачи
- `priority` (Enum) — приоритет задачи: `Низкий`, `Средний`, `Высокий`
- `current_time` (DateTime) — текущее время
- `user_notification_settings` (JSON) — настройки пользователя

**Выходные данные:**
- `reminder_id` (UUID) — идентификатор созданного напоминания
- `trigger_time` (DateTime) — рассчитанное время срабатывания
- `priority_multiplier` (Decimal) — коэффициент приоритета
- `auto_generated` (Boolean) — флаг автоматической генерации

**Бизнес-логика:**
1. Система автоматически создаёт напоминания для задач с дедлайном.
2. Время напоминания рассчитывается на основе приоритета:
   - Высокий: за 24 часа и за 2 часа до дедлайна
   - Средний: за 12 часов до дедлайна
   - Низкий: за 1 час до дедлайна
3. Если у пользователя включены тихие часы, напоминание переносится на начало активного периода.
4. Автоматические напоминания создаются только если пользователь не создал свои.

**Примеры JSON:**

*Входные данные (внутреннее событие при создании/обновлении задачи)*
```json
{
  "task_id": "task_abc123xyz789",
  "due_date": "2024-01-15T10:00:00Z",
  "priority": "Высокий",
  "current_time": "2024-01-14T10:00:00Z",
  "user_notification_settings": {
    "quiet_hours": {"start": "22:00", "end": "08:00"},
    "preferred_channels": ["push"]
  }
}
```

*Выходные данные (автоматически созданное напоминание)*
```json
{
  "reminder_id": "rem_auto123456789",
  "task_id": "task_abc123xyz789",
  "trigger_time": "2024-01-14T10:00:00Z",
  "priority_multiplier": 2.0,
  "auto_generated": true,
  "channels": ["push"]
}
```

**Поэтапная разработка:**
- **Этап 1:** Триггер на создание/обновление задачи с дедлайном
- **Этап 2:** Расчёт времени напоминания по приоритету
- **Этап 3:** Учёт тихих часов при планировании
- **Этап 4:** Проверка на существование пользовательских напоминаний

### 8.3. Фича: Мультиканальная доставка уведомлений

**Входные данные:**
- `reminder_id` (UUID) — идентификатор напоминания
- `channels` (Array) — целевые каналы: `email`, `push`, `sms`
- `message_content` (String) — текст уведомления
- `user_timezone` (String) — часовой пояс пользователя
- `delivery_priority` (Enum) — приоритет доставки: `normal`, `urgent`

**Выходные данные:**
- `delivery_results` (Array) — результаты по каждому каналу
  - `channel` (Enum) — название канала
  - `status` (Enum) — статус: `sent`, `delivered`, `failed`
  - `sent_at` (DateTime) — время отправки
  - `delivered_at` (DateTime) — время доставки (если применимо)
  - `error_message` (String) — сообщение об ошибке (если есть)

**Бизнес-логика:**
1. Отправка производится параллельно во все указанные каналы.
2. Каждый канал имеет своего провайдера и настройку retry-логики.
3. При неудачной отправке делается до 3 попыток с экспоненциальной задержкой.
4. Статус доставки отслеживается для каждого канала отдельно.
5. Канал SMS используется только для критических уведомлений (высокий приоритет).

**Примеры JSON:**

*Входные данные (внутренний запрос на отправку)*
```json
{
  "reminder_id": "rem_xyz789abc123",
  "channels": ["email", "push", "sms"],
  "message_content": "Напоминание: Подготовить презентацию (дедлайн через 1 час)",
  "user_timezone": "UTC+3",
  "delivery_priority": "urgent"
}
```

*Выходные данные (результаты доставки)*
```json
{
  "delivery_results": [
    {
      "channel": "email",
      "status": "delivered",
      "sent_at": "2024-01-15T08:00:00Z",
      "delivered_at": "2024-01-15T08:00:12Z",
      "error_message": null
    },
    {
      "channel": "push",
      "status": "delivered",
      "sent_at": "2024-01-15T08:00:01Z",
      "delivered_at": "2024-01-15T08:00:03Z",
      "error_message": null
    },
    {
      "channel": "sms",
      "status": "failed",
      "sent_at": "2024-01-15T08:00:02Z",
      "delivered_at": null,
      "error_message": "Недостаточно средств на счету SMS-провайдера"
    }
  ]
}
```

**Поэтапная разработка:**
- **Этап 1:** Интеграция с email-провайдером (SendGrid, SES, etc.)
- **Этап 2:** Интеграция с push-сервисом (Firebase, OneSignal)
- **Этап 3:** Интеграция с SMS-провайдером (Twilio, SMS.ru)
- **Этап 4:** Реализация retry-логики и мониторинга доставки

### 8.4. Фича: Умная snooze-функция

**Входные данные:**
- `reminder_id` (UUID) — идентификатор напоминания
- `user_id` (UUID) — идентификатор пользователя
- `snooze_action` (Enum) — действие: `snooze_5min`, `snooze_15min`, `snooze_1h`, `custom`
- `custom_snooze_until` (DateTime) — кастомное время (если выбрано custom)
- `previous_snooze_count` (Integer) — количество предыдущих откладываний

**Выходные данные:**
- `reminder_id` (UUID) — идентификатор напоминания
- `new_trigger_time` (DateTime) — новое время срабатывания
- `snooze_count` (Integer) — общее количество откладываний
- `snooze_limit_reached` (Boolean) — достигнут ли лимит
- `adaptive_interval_used` (Boolean) — использован ли адаптивный интервал

**Бизнес-логика:**
1. Пользователь может отложить напоминание на стандартные интервалы (5, 15, 30 минут, 1 час).
2. Адаптивная система предлагает интервал на основе истории поведения пользователя.
3. Максимальное количество откладываний по умолчанию равно 3 (настраивается).
4. После достижения лимита напоминание становится постоянным до выполнения или отмены.
5. Интервал может увеличиваться при каждом последующем snooze.

**Примеры JSON:**

*Входные данные (POST /api/reminders/{reminder_id}/snooze)*
```json
{
  "reminder_id": "rem_xyz789abc123",
  "user_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "snooze_action": "snooze_15min",
  "previous_snooze_count": 1
}
```

*Выходные данные (ответ на snooze)*
```json
{
  "reminder_id": "rem_xyz789abc123",
  "new_trigger_time": "2024-01-15T08:30:00Z",
  "snooze_count": 2,
  "snooze_limit_reached": false,
  "adaptive_interval_used": false
}
```

**Поэтапная разработка:**
- **Этап 1:** Базовая snooze-функция с фиксированными интервалами
- **Этап 2:** Ограничение максимального количества откладываний
- **Этап 3:** Сбор данных для адаптивного обучения
- **Этап 4:** Рекомендации адаптивных интервалов на основе ML

### 8.5. Фича: Групповые уведомления для командных проектов

**Входные данные:**
- `project_id` (UUID) — идентификатор проекта
- `task_id` (UUID) — идентификатор просроченной задачи
- `assignee_id` (UUID) — идентификатор исполнителя
- `overdue_hours` (Integer) — количество часов просрочки
- `escalation_level` (Enum) — уровень эскалации: `level_1`, `level_2`, `level_3`

**Выходные данные:**
- `notification_id` (UUID) — идентификатор группового уведомления
- `recipients` (Array) — список получателей
  - `user_id` (UUID) — идентификатор пользователя
  - `role` (Enum) — роль в проекте
  - `notification_channel` (Enum) — канал доставки
- `escalation_action` (String) — действие эскалации
- `message` (String) — текст уведомления

**Бизнес-логика:**
1. При просрочке задачи исполнитель получает первое уведомление.
2. Через 24 часа просрочки уведомление отправляется владельцу проекта (эскалация уровня 1).
3. Через 48 часов уведомляются все участники проекта с ролью Редактор и выше (уровень 2).
4. Через 72 часа проект помечается как «проблемный», уведомляется владелец компании (уровень 3).
5. Эскалация останавливается при выполнении задачи.

**Примеры JSON:**

*Входные данные (внутреннее событие эскалации)*
```json
{
  "project_id": "proj_abc123xyz789",
  "task_id": "task_abc123xyz789",
  "assignee_id": "user_def456uvw012",
  "overdue_hours": 48,
  "escalation_level": "level_2"
}
```

*Выходные данные (групповое уведомление)*
```json
{
  "notification_id": "notif_group123456",
  "recipients": [
    {
      "user_id": "user_def456uvw012",
      "role": "Редактор",
      "notification_channel": "push"
    },
    {
      "user_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
      "role": "Владелец",
      "notification_channel": "email"
    }
  ],
  "escalation_action": "notify_all_editors",
  "message": "Задача 'Подготовить презентацию' просрочена на 48 часов. Требуется внимание команды."
}
```

**Поэтапная разработка:**
- **Этап 1:** Детектирование просроченных задач в проектах
- **Этап 2:** Уведомление исполнителя о просрочке
- **Этап 3:** Реализация многоуровневой эскалации
- **Этап 4:** Групповая рассылка участникам проекта

## 7. Модель данных

### 7.1. Пользователь (User)
| Поле | Тип | Описание |
| :--- | :--- | :--- |
| `id` | UUID | Уникальный идентификатор пользователя |
| `email` | String | Электронная почта (уникальная) |
| `password_hash` | String | Хэш пароля (bcrypt) |
| `created_at` | DateTime | Дата регистрации |

### 7.2. Задача (Task)
| Поле | Тип | Описание |
| :--- | :--- | :--- |
| `id` | UUID | Уникальный идентификатор задачи |
| `user_id` | UUID | Идентификатор создателя (FK на User.id) |
| `project_id` | UUID | Идентификатор проекта (FK на Project.id, опционально) |
| `assignee_id` | UUID | Идентификатор исполнителя (FK на User.id, опционально) |
| `title` | String | Название задачи |
| `description` | Text | Подробное описание задачи |
| `status` | Enum | Статус: `Новая`, `В процессе`, `Выполнена` |
| `priority` | Enum | Приоритет: `Низкий`, `Средний`, `Высокий` |
| `due_date` | DateTime | Дедлайн (опционально) |
| `created_at` | DateTime | Дата создания |

### 7.3. Проект (Project)
| Поле | Тип | Описание |
| :--- | :--- | :--- |
| `id` | UUID | Уникальный идентификатор проекта |
| `name` | String | Название проекта |
| `owner_id` | UUID | Владелец проекта (FK на User.id) |

### 7.4. Участники проекта (ProjectMember)
| Поле | Тип | Описание |
| :--- | :--- | :--- |
| `project_id` | UUID | FK на Project.id |
| `user_id` | UUID | FK на User.id |
| `role` | Enum | Роль: `Владелец`, `Редактор`, `Наблюдатель` |

### 7.5. Напоминание (Reminder)
| Поле | Тип | Описание |
| :--- | :--- | :--- |
| `id` | UUID | Уникальный идентификатор напоминания |
| `task_id` | UUID | FK на Task.id |
| `user_id` | UUID | FK на User.id |
| `reminder_type` | Enum | Тип: `Статическое`, `Относительное`, `Умное`, `Эскалационное` |
| `trigger_time` | DateTime | Время срабатывания напоминания |
| `offset_minutes` | Integer | Смещение в минутах от дедлайна (для относительных) |
| `channels` | JSON | Массив каналов доставки |
| `status` | Enum | Статус: `Активное`, `Отложено`, `Выполнено`, `Отменено` |
| `attempts_count` | Integer | Количество попыток отправки |
| `max_attempts` | Integer | Максимальное количество попыток |
| `last_sent_at` | DateTime | Время последней отправки |
| `snooze_until` | DateTime | Отложить до (для snooze-функции) |
| `effectiveness_score` | Decimal | Оценка эффективности (0-10) |
| `created_at` | DateTime | Дата создания |

### 7.6. Настройки уведомлений (NotificationSettings)
| Поле | Тип | Описание |
| :--- | :--- | :--- |
| `user_id` | UUID | FK на User.id (Primary Key) |
| `quiet_hours_start` | Time | Начало тихих часов |
| `quiet_hours_end` | Time | Конец тихих часов |
| `timezone` | String | Часовой пояс пользователя |
| `preferred_channels` | JSON | Предпочитаемые каналы уведомлений |
| `smart_scheduling_enabled` | Boolean | Включено ли умное планирование |
| `batch_notifications` | Boolean | Группировка уведомлений |
| `escalation_enabled` | Boolean | Включена ли эскалация |
| `updated_at` | DateTime | Дата последнего обновления |

### 7.7. Лог уведомлений (NotificationLog)
| Поле | Тип | Описание |
| :--- | :--- | :--- |
| `id` | UUID | Уникальный идентификатор записи |
| `reminder_id` | UUID | FK на Reminder.id |
| `channel` | Enum | Канал: `Email`, `Push`, `SMS` |
| `status` | Enum | Статус: `Отправлено`, `Доставлено`, `Ошибка`, `Открыто`, `Нажато` |
| `sent_at` | DateTime | Время отправки |
| `delivered_at` | DateTime | Время доставки |
| `opened_at` | DateTime | Время открытия |
| `clicked_at` | DateTime | Время клика |
| `error_message` | Text | Сообщение об ошибке (если есть) |

### 7.8. Паттерны активности (UserActivityPattern)
| Поле | Тип | Описание |
| :--- | :--- | :--- |
| `user_id` | UUID | FK на User.id |
| `hour_of_day` | Integer | Час дня (0-23) |
| `day_of_week` | Integer | День недели (1-7) |
| `activity_score` | Decimal | Оценка активности (0-1) |
| `completion_rate` | Decimal | Процент выполнения задач |
| `response_time_avg` | Integer | Среднее время отклика в минутах |
| `last_updated` | DateTime | Дата последнего обновления |

### 7.9. Связи
*   **Один-ко-многим:** Один `Пользователь` может быть владельцем множества `Проектов`.
*   **Многие-ко-многим:** `Пользователи` могут участвовать во многих `Проектах` через таблицу `ProjectMember`.
*   **Иерархия:** `Задачи` могут принадлежать `Проекту` или быть личными (при `project_id = NULL`).
