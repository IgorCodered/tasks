# Техническое задание: TODO Manager

## 1. Общие сведения
**TODO Manager** — это веб-приложение для управления личными задачами, позволяющее пользователям создавать, классифицировать и отслеживать выполнение дел.

## 2. Бизнес-правила
1. **Авторизация:** Доступ к задачам осуществляется только после регистрации и входа в систему.
2. **Изоляция данных:** Пользователь видит только свои задачи.
3. **Статус задачи:** Задача может находиться в одном из трех состояний: `Новая`, `В процессе`, `Выполнена`.
4. **Приоритеты:** Каждая задача имеет уровень приоритета: `Низкий`, `Средний`, `Высокий`.
5. **Удаление:** Удаление задачи является безвозвратным.

## 3. Функциональные требования (ФТ)

### ФТ-1: Регистрация и аутентификация пользователей (JWT)

**Для чего нужна эта фича:** Обеспечение безопасного доступа пользователей к системе и создание персональных учётных записей.

**Какую проблему она решает:** Защищает данные пользователей от несанкционированного доступа, гарантирует изоляцию личных задач между разными пользователями и позволяет идентифицировать пользователя для персонализации опыта.

**Дополнительное описание:** Система использует JWT-токены для Stateless-аутентификации. При регистрации пользователь указывает email и пароль, которые проходят валидацию. Пароль хэшируется алгоритмом bcrypt перед сохранением. После входа пользователь получает access-токен и refresh-токен. Access-токен используется для авторизации запросов к API, refresh-токен — для обновления access-токена без повторного ввода пароля. Токены имеют ограниченное время жизни.

*   **Входные данные:**
    - `email` (String) — электронная почта пользователя
    - `password` (String) — пароль (минимум 8 символов)
    - `refresh_token` (String) — токен обновления (для refresh-операции)

*   **Выходные данные:**
    - `user_id` (UUID) — идентификатор пользователя
    - `access_token` (String) — JWT-токен доступа
    - `refresh_token` (String) — JWT-токен обновления
    - `expires_in` (Integer) — время жизни токена в секундах

*   **Бизнес-логика:**
    1. Email должен быть уникальным в системе.
    2. Пароль должен содержать минимум 8 символов, включая буквы и цифры.
    3. Access-токен действителен 15 минут.
    4. Refresh-токен действителен 7 дней.
    5. При каждом новом входе все предыдущие сессии пользователя аннулируются.
    6. Неверный пароль более 5 раз подряд блокирует учётную запись на 30 минут.

*   **Примеры JSON:**

*Входные данные (POST /api/auth/register)*
```json
{
  "email": "user@example.com",
  "password": "SecurePass123"
}
```

*Выходные данные (ответ на успешную регистрацию/вход)*
```json
{
  "user_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refresh_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "expires_in": 900
}
```

*   **Поэтапная разработка:**
    - **Этап 1:** Реализация эндпоинтов регистрации и входа
    - **Этап 2:** Интеграция JWT для генерации и валидации токенов
    - **Этап 3:** Механизм refresh-токенов для продления сессии
    - **Этап 4:** Защита от brute-force атак и управление блокировками

### ФТ-2: Создание задачи

**Для чего нужна эта фича:** Позволяет пользователям фиксировать дела, которые необходимо выполнить, с указанием ключевых параметров для организации работы.

**Какую проблему она решает:** Устраняет необходимость держать задачи в голове или на бумажных носителях, обеспечивает структурированное хранение информации о делах с возможностью приоритизации и планирования сроков.

**Дополнительное описание:** Пользователь создаёт задачу с обязательным названием и опциональными описанием, дедлайном и приоритетом. Задача автоматически получает статус «Новая» и привязывается к пользователю-создателю. После создания задача появляется в общем списке и может быть отфильтрована по различным критериям.

*   **Входные данные:**
    - `title` (String) — название задачи (обязательно, 1–200 символов)
    - `description` (String) — подробное описание (опционально, до 5000 символов)
    - `due_date` (DateTime) — дедлайн выполнения (опционально)
    - `priority` (Enum) — приоритет: `Низкий`, `Средний`, `Высокий` (по умолчанию `Средний`)
    - `project_id` (UUID) — идентификатор проекта (опционально)

*   **Выходные данные:**
    - `task_id` (UUID) — идентификатор созданной задачи
    - `title` (String) — название задачи
    - `status` (Enum) — статус задачи (`Новая`)
    - `created_at` (DateTime) — дата и время создания

*   **Бизнес-логика:**
    1. Название задачи обязательно и должно быть уникальным в пределах одного дня для пользователя.
    2. Дедлайн не может быть установлен в прошлом.
    3. При создании задачи без указания приоритета устанавливается значение по умолчанию (`Средний`).
    4. Задача автоматически получает статус «Новая».
    5. Пользователь может создать неограниченное количество задач.
    6. Если указан project_id, пользователь должен быть участником этого проекта.

*   **Примеры JSON:**

*Входные данные (POST /api/tasks)*
```json
{
  "title": "Подготовить презентацию для клиента",
  "description": "Собрать данные по проекту X, создать слайды с метриками и выводами",
  "due_date": "2024-01-20T18:00:00Z",
  "priority": "Высокий",
  "project_id": "proj_abc123xyz789"
}
```

*Выходные данные (ответ на создание задачи)*
```json
{
  "task_id": "task_abc123xyz789",
  "title": "Подготовить презентацию для клиента",
  "status": "Новая",
  "priority": "Высокий",
  "due_date": "2024-01-20T18:00:00Z",
  "created_at": "2024-01-15T10:30:00Z"
}
```

*   **Поэтапная разработка:**
    - **Этап 1:** CRUD операции для задач (создание, чтение)
    - **Этап 2:** Валидация входных данных и установка значений по умолчанию
    - **Этап 3:** Привязка задач к проектам
    - **Этап 4:** Оптимизация производительности при большом количестве задач

### ФТ-3: Просмотр списка задач с фильтрацией

**Для чего нужна эта фича:** Предоставление пользователю удобного интерфейса для просмотра всех своих задач с возможностью сортировки и фильтрации по различным критериям.

**Какую проблему она решает:** Позволяет быстро находить нужные задачи среди большого количества дел, фокусироваться на приоритетных или срочных задачах, отслеживать прогресс выполнения.

**Дополнительное описание:** Пользователь может просматривать задачи в виде списка с поддержкой пагинации. Доступны фильтры по статусу, приоритету, наличию дедлайна, принадлежности к проекту. Поддерживается сортировка по дате создания, дедлайну, приоритету, названию. Поиск осуществляется по названию и описанию задач.

*   **Входные данные:**
    - `page` (Integer) — номер страницы (по умолчанию 1)
    - `page_size` (Integer) — количество задач на странице (по умолчанию 20, максимум 100)
    - `status` (Enum) — фильтр по статусу (опционально)
    - `priority` (Enum) — фильтр по приоритету (опционально)
    - `project_id` (UUID) — фильтр по проекту (опционально)
    - `search_query` (String) — поисковый запрос по названию и описанию (опционально)
    - `sort_by` (Enum) — поле для сортировки: `created_at`, `due_date`, `priority`, `title`
    - `sort_order` (Enum) — порядок сортировки: `asc`, `desc`

*   **Выходные данные:**
    - `tasks` (Array) — массив задач
    - `total_count` (Integer) — общее количество задач
    - `page` (Integer) — текущая страница
    - `page_size` (Integer) — размер страницы
    - `total_pages` (Integer) — общее количество страниц

*   **Бизнес-логика:**
    1. Пользователь видит только свои задачи и задачи проектов, где он является участником.
    2. Фильтры могут комбинироваться (например, статус + приоритет).
    3. Поиск по query выполняется по подстроке в названии и описании (case-insensitive).
    4. Сортировка по приоритету учитывает вес: Высокий > Средний > Низкий.
    5. Задачи с истёкшим дедлайном визуально выделяются (флаг overdue).
    6. Максимальный размер страницы ограничен 100 задачами для производительности.

*   **Примеры JSON:**

*Входные данные (GET /api/tasks?page=1&status=В процессе&priority=Высокий&sort_by=due_date)*
```json
{}
```

*Выходные данные (ответ со списком задач)*
```json
{
  "tasks": [
    {
      "task_id": "task_abc123xyz789",
      "title": "Подготовить презентацию для клиента",
      "status": "В процессе",
      "priority": "Высокий",
      "due_date": "2024-01-20T18:00:00Z",
      "created_at": "2024-01-15T10:30:00Z",
      "overdue": false
    },
    {
      "task_id": "task_def456uvw012",
      "title": "Отчёт по продажам",
      "status": "В процессе",
      "priority": "Высокий",
      "due_date": "2024-01-14T12:00:00Z",
      "created_at": "2024-01-10T09:00:00Z",
      "overdue": true
    }
  ],
  "total_count": 47,
  "page": 1,
  "page_size": 20,
  "total_pages": 3
}
```

*   **Поэтапная разработка:**
    - **Этап 1:** Базовый список задач без фильтрации
    - **Этап 2:** Реализация фильтров по статусу и приоритету
    - **Этап 3:** Полнотекстовый поиск по названию и описанию
    - **Этап 4:** Сортировка и пагинация результатов

### ФТ-4: Редактирование и изменение статуса задачи

**Для чего нужна эта фича:** Позволяет пользователю актуализировать информацию о задаче и отслеживать прогресс её выполнения через изменение статуса.

**Какую проблему она решает:** Обеспечивает гибкость управления задачами — можно корректировать сроки, приоритеты, описание при изменении обстоятельств, а также фиксировать этапы выполнения через статусы.

**Дополнительное описание:** Пользователь может редактировать любые поля задачи кроме идентификатора. Изменение статуса фиксируется в истории задачи. При переходе в статус «Выполнена» автоматически проставляется дата завершения. Редактирование доступно только автору задачи или участникам проекта с соответствующими правами.

*   **Входные данные:**
    - `task_id` (UUID) — идентификатор задачи
    - `title` (String) — новое название (опционально)
    - `description` (String) — новое описание (опционально)
    - `due_date` (DateTime) — новый дедлайн (опционально)
    - `priority` (Enum) — новый приоритет (опционально)
    - `status` (Enum) — новый статус (опционально)

*   **Выходные данные:**
    - `task_id` (UUID) — идентификатор задачи
    - `updated_fields` (Array) — список изменённых полей
    - `updated_at` (DateTime) — дата и время последнего изменения
    - `previous_status` (Enum) — предыдущий статус (если менялся)
    - `current_status` (Enum) — текущий статус

*   **Бизнес-логика:**
    1. Изменять задачу может только её создатель или участники проекта с ролью «Редактор» или «Владелец».
    2. При изменении статуса с любого на «Выполнена» фиксируется completed_at.
    3. Нельзя изменить статус выполненной задачи на «Новая» или «В процессе» (только через снятие выполнения).
    4. История изменений статуса сохраняется для аналитики.
    5. Дедлайн можно переносить неограниченное количество раз.
    6. При изменении приоритета на «Высокий» для просроченной задачи создаётся уведомление.

*   **Примеры JSON:**

*Входные данные (PATCH /api/tasks/{task_id})*
```json
{
  "status": "Выполнена",
  "priority": "Средний"
}
```

*Выходные данные (ответ на обновление)*
```json
{
  "task_id": "task_abc123xyz789",
  "updated_fields": ["status", "priority"],
  "updated_at": "2024-01-15T16:45:00Z",
  "previous_status": "В процессе",
  "current_status": "Выполнена",
  "completed_at": "2024-01-15T16:45:00Z"
}
```

*   **Поэтапная разработка:**
    - **Этап 1:** API для обновления полей задачи
    - **Этап 2:** Логирование изменений статуса
    - **Этап 3:** Валидация прав доступа при редактировании
    - **Этап 4:** Автоматическое проставление даты завершения

### ФТ-5: Удаление задачи

**Для чего нужна эта фича:** Позволяет пользователю безвозвратно удалить задачу, которая больше не актуальна или была создана ошибочно.

**Какую проблему она решает:** Очищает список задач от ненужной информации, предотвращает накопление «мусорных» задач, даёт пользователю полный контроль над своими данными.

**Дополнительное описание:** Удаление задачи является необратимой операцией. Перед удалением система проверяет права пользователя. Все связанные данные (напоминания, история статусов, назначения) удаляются каскадно. Удалённые задачи не попадают в аналитику.

*   **Входные данные:**
    - `task_id` (UUID) — идентификатор удаляемой задачи
    - `confirmation` (Boolean) — подтверждение удаления (должно быть true)

*   **Выходные данные:**
    - `task_id` (UUID) — идентификатор удалённой задачи
    - `deleted_at` (DateTime) — дата и время удаления
    - `success` (Boolean) — флаг успешного удаления

*   **Бизнес-логика:**
    1. Удалять задачу может только её создатель или владелец проекта.
    2. Удаление необратимо — восстановлению задача не подлежит.
    3. Все связанные напоминания и уведомления удаляются каскадно.
    4. Если задача назначена на исполнителя, он получает уведомление об удалении.
    5. Удалённые задачи исключаются из расчёта метрик аналитики.
    6. Операция удаления логируется для аудита.

*   **Примеры JSON:**

*Входные данные (DELETE /api/tasks/{task_id})*
```json
{
  "confirmation": true
}
```

*Выходные данные (ответ на удаление)*
```json
{
  "task_id": "task_abc123xyz789",
  "deleted_at": "2024-01-15T17:00:00Z",
  "success": true
}
```

*   **Поэтапная разработка:**
    - **Этап 1:** API для удаления задачи
    - **Этап 2:** Каскадное удаление связанных данных
    - **Этап 3:** Валидация прав доступа
    - **Этап 4:** Логирование операций удаления

### ФТ-6: Поиск задач по ключевым словам

**Для чего нужна эта фича:** Позволяет пользователю быстро находить задачи по содержимому названия или описания без необходимости применять сложные фильтры.

**Какую проблему она решает:** Экономит время пользователя при работе с большим количеством задач, устраняет необходимость ручного просмотра списка для нахождения конкретной задачи.

**Дополнительное описание:** Поиск осуществляется по подстроке в полях title и description. Поддерживается нечёткий поиск (fuzzy search) для учёта опечаток. Результаты ранжируются по релевантности. Поиск работает в реальном времени с минимальной задержкой.

*   **Входные данные:**
    - `query` (String) — поисковый запрос (минимум 2 символа)
    - `limit` (Integer) — максимальное количество результатов (по умолчанию 10)

*   **Выходные данные:**
    - `results` (Array) — массив найденных задач
    - `query` (String) — исходный поисковый запрос
    - `total_found` (Integer) — общее количество найденных задач

*   **Бизнес-логика:**
    1. Поисковый запрос должен содержать минимум 2 символа.
    2. Поиск регистронезависимый (case-insensitive).
    3. Результаты ранжируются по релевантности (совпадение в названии имеет больший вес).
    4. Поиск выполняется только по задачам текущего пользователя.
    5. Поддерживается поиск по нескольким словам (AND-логика).
    6. Время ответа поиска не должно превышать 100 мс.

*   **Примеры JSON:**

*Входные данные (GET /api/tasks/search?query=презентация клиент)*
```json
{}
```

*Выходные данные (результаты поиска)*
```json
{
  "results": [
    {
      "task_id": "task_abc123xyz789",
      "title": "Подготовить презентацию для клиента",
      "description": "Собрать данные по проекту X, создать слайды",
      "relevance_score": 0.95
    },
    {
      "task_id": "task_ghi789jkl012",
      "title": "Встреча с клиентом",
      "description": "Обсудить презентацию и дальнейшие шаги",
      "relevance_score": 0.72
    }
  ],
  "query": "презентация клиент",
  "total_found": 2
}
```

*   **Поэтапная разработка:**
    - **Этап 1:** Базовый поиск по подстроке в названии
    - **Этап 2:** Добавление поиска по описанию
    - **Этап 3:** Ранжирование результатов по релевантности
    - **Этап 4:** Оптимизация производительности поиска (индексы)

## 4. Нефункциональные требования (НФТ)
*   **НФТ-1 (Производительность):** Время отклика API не должно превышать 200 мс при стандартной нагрузке.
*   **НФТ-2 (Безопасность):** Пароли пользователей должны храниться в зашифрованном виде (bcrypt).
*   **НФТ-3 (Масштабируемость):** Архитектура должна поддерживать до 10,000 активных пользователей.
*   **НФТ-4 (Доступность):** Приложение должно быть доступно 99.9% времени (Uptime).
*   **НФТ-5 (Уведомления):** Система должна гарантировать доставку критических напоминаний в течение 30 секунд.
*   **НФТ-6 (Персонализация):** Алгоритм машинного обучения должен адаптироваться к паттернам пользователя за 14 дней использования.
*   **НФТ-7 (Надежность):** Система должна иметь fallback-механизмы при недоступности основных каналов уведомлений.

### 5.1. Фича: Трекинг Daily Active Users (DAU)

**Для чего нужна эта фича:** Измерение ежедневной активности пользователей для оценки вовлечённости и популярности приложения.

**Какую проблему она решает:** Позволяет продукт-менеджерам и команде понимать, сколько уникальных пользователей взаимодействуют с приложением каждый день, выявлять тренды роста или спада активности, оценивать эффективность новых функций и маркетинговых кампаний.

**Дополнительное описание:** Система автоматически собирает события активности пользователей при каждом действии в приложении. DAU рассчитывается как количество уникальных пользователей, совершивших хотя бы одно действие за календарный день. Метрика агрегируется ежедневно и доступна через API для построения дашбордов и отчётов.

**Входные данные:**
- `user_id` (UUID) — идентификатор пользователя
- `session_id` (UUID) — идентификатор сессии
- `timestamp` (DateTime) — время активности
- `action_type` (String) — тип действия (login, view_task, create_task, complete_task, etc.)
- `device_info` (JSON) — информация об устройстве

**Выходные данные:**
- `date` (Date) — дата отчётности
- `dau_count` (Integer) — количество уникальных пользователей
- `dau_by_action` (JSON) — разбивка по типам действий
- `dau_trend` (Array) — динамика за последние 7 дней

**Бизнес-логика:**
1. Пользователь считается активным, если выполнил хотя бы одно действие в течение календарного дня (00:00–23:59 по UTC).
2. Повторные действия одного пользователя в течение дня учитываются один раз.
3. Действия анонимных пользователей не учитываются.
4. Расчёт производится ежедневно в 00:05 UTC за предыдущий день.

**Примеры JSON:**

*Входные данные (POST /api/analytics/events)*
```json
{

  "action_type": "complete_task",
  "device_info": {
    "platform": "web",
    "browser": "Chrome",
    "os": "Windows"
  }
}
```

*Выходные данные (GET /api/analytics/dau?date=2024-01-15)*
```json
{
  "date": "2024-01-15",
  "dau_count": 1247,
  "dau_by_action": {
    "login": 892,
    "view_task": 1105,
    "create_task": 456,
    "complete_task": 378,
    "delete_task": 89
  },
  "dau_trend": [
    {"date": "2024-01-09", "count": 1189},
    {"date": "2024-01-10", "count": 1234},
    {"date": "2024-01-11", "count": 1156},
```

- **Этап 3:** Построение трендов и визуализация динамики
- **Этап 4:** Оптимизация производительности для больших объёмов данных

### 5.2. Фича: Task Completion Rate

**Для чего нужна эта фича:** Измерение процента завершённых задач от общего количества созданных для оценки продуктивности пользователей.

**Какую проблему она решает:** Помогает понять, насколько эффективно пользователи справляются со своими задачами, выявить проблемы с прокрастинацией или перегрузкой, оценить влияние новых функций на завершаемость задач.

**Дополнительное описание:** Метрика рассчитывается за различные периоды (7 дней, 30 дней, всё время) и может сегментироваться по приоритетам задач. Система автоматически отслеживает изменения статуса задач и вычисляет среднее время выполнения.

**Входные данные:**
- `task_id` (UUID) — идентификатор задачи
- `user_id` (UUID) — идентификатор пользователя
- `initial_status` (Enum) — начальный статус задачи
- `final_status` (Enum) — конечный статус задачи
- `status_changed_at` (DateTime) — время изменения статуса
- `created_at` (DateTime) — время создания задачи

**Выходные данные:**
- `period` (String) — период отчёта (last_7_days, last_30_days, all_time)
- `total_tasks_created` (Integer) — всего создано задач
- `total_tasks_completed` (Integer) — всего завершено задач
- `completion_rate` (Decimal) — процент завершения (0–1)
- `completion_rate_by_priority` (JSON) — разбивка по приоритетам
- `avg_completion_time_hours` (Decimal) — среднее время выполнения в часах

**Бизнес-логика:**
1. Задача считается завершённой, если её статус изменён на «Выполнена».
2. В расчёт включаются только задачи, созданные в пределах отчётного периода.
3. Если задача была создана и завершена в одном периоде, она учитывается в числителе и знаменателе.
4. Задачи со статусом «Новая» или «В процессе» не учитываются в числителе.
5. Расчёт производится по формуле: `(кол-во завершённых задач) / (кол-во созданных задач)`.

**Примеры JSON:**

*Входные данные (внутреннее событие изменения статуса)*
```json
{
  "task_id": "task_abc123xyz789",
  "user_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "initial_status": "В процессе",
  "final_status": "Выполнена",
  "status_changed_at": "2024-01-15T16:45:00Z",
  "created_at": "2024-01-14T10:00:00Z"
}
```

*Выходные данные (GET /api/analytics/task-completion?period=last_30_days)*
```json
{
  "period": "last_30_days",
  "total_tasks_created": 542,
  "total_tasks_completed": 387,
  "completion_rate": 0.71,
  "completion_rate_by_priority": {
    "Низкий": 0.65,
    "Средний": 0.72,
    "Высокий": 0.84
  },
  "avg_completion_time_hours": 18.5
}
```

**Поэтапная разработка:**
- **Этап 1:** Логирование изменений статуса задач
- **Этап 2:** Расчёт completion rate за фиксированный период
- **Этап 3:** Сегментация по приоритетам и другим параметрам
- **Этап 4:** Расчёт среднего времени выполнения задач

### 5.3. Фича: Retention Rate

**Для чего нужна эта фича:** Оценка способности приложения удерживать пользователей в течение длительного времени.

**Какую проблему она решает:** Позволяет определить, насколько хорошо приложение вовлекает новых пользователей, выявить проблемы с оттоком на ранних этапах использования, оценить долгосрочную ценность продукта для пользователей.

**Дополнительное описание:** Retention Rate рассчитывается по когортам пользователей (группам по дате регистрации) и показывает процент пользователей, вернувшихся в приложение через определённое количество дней после регистрации. Система строит кривую удержания для визуализации динамики.

**Входные данные:**
- `user_id` (UUID) — идентификатор пользователя
- `registration_date` (Date) — дата регистрации
- `activity_dates` (Array of Date) — даты активности пользователя
- `cohort` (String) — когорта пользователя (неделя/месяц регистрации)

**Выходные данные:**
- `cohort_period` (String) — период когорты (например, 2024-W02)
- `registered_users` (Integer) — количество зарегистрированных пользователей в когорте
- `retention_day_1` (Decimal) — процент вернувшихся через 1 день
- `retention_day_7` (Decimal) — процент вернувшихся через 7 дней
- `retention_day_30` (Decimal) — процент вернувшихся через 30 дней
- `retention_curve` (Array) — кривая удержания по дням

**Бизнес-логика:**
1. Пользователь считается вернувшимся, если он проявил активность в целевой день (см. логику DAU).
2. Когорта формируется по неделе регистрации пользователя.
3. Retention Rate рассчитывается как `(кол-во вернувшихся пользователей) / (кол-во пользователей в когорте)`.
4. День 0 — это день регистрации.
5. Расчёт производится еженедельно для всех активных когорт.

**Примеры JSON:**

*Входные данные (агрегированные данные по пользователю)*
```json

  "user_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",

  "activity_dates": ["2024-01-08", "2024-01-09", "2024-01-15", "2024-01-22"],
  "cohort": "2024-W02"
}
```

*Выходные данные (GET /api/analytics/retention?cohort=2024-W02)*
```json
{
  "cohort_period": "2024-W02",
  "registered_users": 342,
  "retention_day_1": 0.68,
  "retention_day_7": 0.45,
  "retention_day_30": 0.28,
  "retention_curve": [
    {"day": 0, "rate": 1.0},
    {"day": 1, "rate": 0.68},
    {"day": 2, "rate": 0.59},
    {"day": 3, "rate": 0.54},
    {"day": 4, "rate": 0.51},
    {"day": 5, "rate": 0.48},
    {"day": 6, "rate": 0.46},
    {"day": 7, "rate": 0.45}
  ]
}
```

**Поэтапная разработка:**
- **Этап 1:** Формирование когорт по дате регистрации
- **Этап 2:** Отслеживание активности пользователей по дням
- **Этап 3:** Расчёт retention для ключевых точек (D1, D7, D30)
- **Этап 4:** Построение полной кривой удержания

### 5.4. Фича: Notification Effectiveness Rate

**Для чего нужна эта фича:** Оценка эффективности системы уведомлений по влиянию на выполнение задач.

**Какую проблему она решает:** Помогает определить, насколько уведомления действительно мотивируют пользователей выполнять задачи, выявить наиболее эффективные каналы доставки и оптимальное время отправки, обосновать инвестиции в инфраструктуру уведомлений.

**Дополнительное описание:** Метрика измеряет процент задач, выполненных в течение 24 часов после получения уведомления. Система сравнивает эффективность разных каналов (email, push, SMS) и формирует рекомендации по выбору канала для каждого пользователя.

**Входные данные:**
- `reminder_id` (UUID) — идентификатор напоминания
- `task_id` (UUID) — идентификатор задачи
- `user_id` (UUID) — идентификатор пользователя
- `notification_sent_at` (DateTime) — время отправки уведомления
- `notification_delivered_at` (DateTime) — время доставки
- `task_completed_at` (DateTime) — время выполнения задачи (если применимо)
- `channel` (Enum) — канал уведомления (email, push, sms)

**Выходные данные:**
- `period` (String) — период отчёта
- `total_notifications_sent` (Integer) — всего отправлено уведомлений
- `tasks_completed_after_notification` (Integer) — задач выполнено после уведомления
- `effectiveness_rate` (Decimal) — процент эффективности (0–1)
- `effectiveness_by_channel` (JSON) — эффективность по каналам
- `avg_time_to_complete_minutes` (Decimal) — среднее время до выполнения

**Бизнес-логика:**
1. Уведомление считается эффективным, если задача была выполнена в течение 24 часов после доставки уведомления.
2. Если по задаче было отправлено несколько уведомлений, учитывается первое, после которого задача была выполнена.
3. Задачи, выполненные до отправки уведомления, не учитываются.
4. Расчёт по формуле: `(задач выполнено после уведомления) / (всего уведомлений доставлено)`.

```json
{
  "period": "last_30_days",
  "total_notifications_sent": 2847,
  "tasks_completed_after_notification": 2195,
```

- **Этап 4:** Анализ времени реакции на уведомления

### 5.5. Фича: Average Response Time

**Для чего нужна эта фича:** Измерение скорости реакции пользователей на уведомления для оценки их своевременности и релевантности.

**Какую проблему она решает:** Помогает определить оптимальное время отправки уведомлений, выявить каналы с fastest response time, понять, какие типы уведомлений пользователи игнорируют или быстро обрабатывают.

**Дополнительное описание:** Система измеряет время между доставкой уведомления и первым действием пользователя (открытие, клик, выполнение задачи). Метрика рассчитывается отдельно для каждого канала и типа действия, что позволяет оптимизировать стратегию уведомлений.

**Входные данные:**
- `reminder_id` (UUID) — идентификатор напоминания
- `notification_delivered_at` (DateTime) — время доставки уведомления
- `user_action_at` (DateTime) — время действия пользователя (открытие, клик, выполнение задачи)
- `action_type` (Enum) — тип действия (open, click, complete_task, snooze)

- `response_time_distribution` (JSON) — распределение по временным интервалам

**Бизнес-логика:**
1. Время отклика измеряется как разница между доставкой уведомления и первым действием пользователя.
2. Если пользователь не совершил действие в течение 24 часов, уведомление исключается из расчёта.
3. Действия snooze (отложить) также учитываются как отклик.
4. Расчёт производится отдельно для каждого канала уведомлений.

### 5.6. Фича: Snooze Rate

**Для чего нужна эта фича:** Анализ частоты откладывания напоминаний пользователями для оценки релевантности времени уведомлений.

**Какую проблему она решает:** Помогает выявить, насколько точно система выбирает время напоминаний, определить паттерны прокрастинации пользователей, оптимизировать алгоритмы умного планирования уведомлений.

**Дополнительное описание:** Snooze Rate показывает процент напоминаний, которые пользователи откладывают вместо выполнения задачи. Высокий snooze rate может указывать на неправильное время отправки или перегрузку пользователя задачами. Система анализирует паттерны по времени суток и дням недели.

**Входные данные:**
- `reminder_id` (UUID) — идентификатор напоминания
- `task_id` (UUID) — идентификатор задачи
- `user_id` (UUID) — идентификатор пользователя
- `original_trigger_time` (DateTime) — оригинальное время срабатывания
- `snooze_until` (DateTime) — время повторного срабатывания
- `snooze_count` (Integer) — количество откладываний данной задачи

**Выходные данные:**
- `period` (String) — период отчёта
- `total_reminders_triggered` (Integer) — всего сработавших напоминаний
- `total_snoozed_reminders` (Integer) — всего отложенных напоминаний
- `snooze_rate` (Decimal) — процент откладываний (0–1)
- `avg_snooze_count_per_task` (Decimal) — среднее кол-во откладываний на задачу
- `snooze_by_time_of_day` (JSON) — частота откладываний по времени суток

**Бизнес-логика:**
1. Напоминание считается отложенным, если пользователь активировал функцию snooze.
2. Максимальное количество откладываний одной задачи ограничено (настраиваемый параметр, по умолчанию 3).
3. Snooze Rate рассчитывается как `(отложенных напоминаний) / (всего сработавших напоминаний)`.
4. Интервал откладывания может быть фиксированным (5, 15, 30 минут) или адаптивным.

**Примеры JSON:**

*Входные данные (событие откладывания напоминания)*
```json
{
  "reminder_id": "rem_xyz789abc123",
  "task_id": "task_abc123xyz789",
  "user_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "original_trigger_time": "2024-01-15T08:00:00Z",
```

### 5.7. Фича: Channel Performance

**Для чего нужна эта фича:** Сравнение эффективности различных каналов доставки уведомлений для оптимизации коммуникационной стратегии.

**Какую проблему она решает:** Позволяет определить, какие каналы (email, push, SMS) наиболее эффективны для разных типов пользователей и задач, обосновать расходы на платные каналы (SMS), сформировать персонализированные рекомендации по каналам.

**Дополнительное описание:** Система отслеживает доставку, открытия, клики и конверсию в выполнение задачи для каждого канала. На основе этих данных формируется рейтинг каналов и рекомендации по выбору оптимального канала для каждого пользователя.

**Входные данные:**
- `reminder_id` (UUID) — идентификатор напоминания
- `channel` (Enum) — канал уведомления (email, push, sms)
- `sent_at` (DateTime) — время отправки
- `delivered_at` (DateTime) — время доставки (если применимо)
- `opened_at` (DateTime) — время открытия (для email)
- `clicked_at` (DateTime) — время клика
- `task_completed_at` (DateTime) — время выполнения задачи

**Выходные данные:**
- `period` (String) — период отчёта
- `channels` (Array) — массив данных по каналам
    - `channel_name` (String) — название канала

    - `delivered_count` (Integer) — доставлено
    - `delivery_rate` (Decimal) — процент доставки
    - `open_rate` (Decimal) — процент открытий (для email)
    - `click_rate` (Decimal) — процент кликов
    - `conversion_rate` (Decimal) — процент выполнения задач

**Бизнес-логика:**
1. Delivery Rate рассчитывается только для каналов с подтверждением доставки (push, sms).
2. Open Rate применим только для email-уведомлений.
3. Conversion Rate — процент задач, выполненных в течение 24 часов после клика/открытия.
4. Каналы ранжируются по conversion rate для формирования рекомендаций.

### 5.8. Фича: Optimal Timing Score

**Для чего нужна эта фича:** Персонализированное определение оптимального времени отправки уведомлений для каждого пользователя на основе его поведения.

**Какую проблему она решает:** Устраняет проблему отправки уведомлений в неподходящее время, когда пользователь не готов к действию, повышает эффективность напоминаний за счёт учёта индивидуальных паттернов активности, снижает раздражение от нерелевантных уведомлений.

**Дополнительное описание:** Система анализирует исторические данные о выполнении задач после напоминаний, выявляет временные слоты с highest completion rate и response rate. На основе этого формируются персонализированные рекомендации по времени отправки для каждого пользователя. Рекомендации обновляются еженедельно.

**Входные данные:**
- `user_id` (UUID) — идентификатор пользователя
- `reminder_id` (UUID) — идентификатор напоминания
- `scheduled_time` (DateTime) — запланированное время напоминания
- `actual_completion_time` (DateTime) — фактическое время выполнения
- `hour_of_day` (Integer) — час суток (0–23)
- `day_of_week` (Integer) — день недели (1–7)
- `is_effective` (Boolean) — было ли напоминание эффективным

**Выходные данные:**
- `user_id` (UUID) — идентификатор пользователя
- `optimal_time_slots` (Array) — рекомендованные временные слоты
    - `time_range` (String) — диапазон времени (например, "08:00-10:00")
    - `score` (Decimal) — оценка эффективности (0–10)
    - `sample_size` (Integer) — количество наблюдений
- `best_day_of_week` (Integer) — лучший день недели
- `confidence_level` (Decimal) — уровень достоверности рекомендации (0–1)

**Бизнес-логика:**
1. Система анализирует исторические данные о выполнении задач после напоминаний.
2. Временной слот считается эффективным, если tasks completed within 2 hours >= 70%.
3. Минимальный размер выборки для рекомендации — 10 наблюдений на слот.
4. Оценка рассчитывается как взвешенная средняя effectiveness rate и response time.
5. Рекомендации обновляются еженедельно на основе новых данных.

**Примеры JSON:**

*Входные данные (агрегированные данные по активности)*

### 6.2. Фича: Назначение исполнителей на задачи

**Для чего нужна эта фича:** Распределение ответственности за выполнение задач между участниками проекта.

**Какую проблему она решает:** Устраняет неопределённость в том, кто отвечает за задачу, позволяет владельцам проектов делегировать работу, обеспечивает прозрачность распределения нагрузки в команде, упрощает отслеживание прогресса.

**Дополнительное описание:** Пользователи с правами Редактора или Владельца могут назначать исполнителей на задачи из числа участников проекта. При назначении исполнитель получает уведомление. История назначений сохраняется для аудита. Одна задача может иметь только одного исполнителя в момент времени.

**Входные данные:**
- `task_id` (UUID) — идентификатор задачи
- `project_id` (UUID) — идентификатор проекта
- `assignee_id` (UUID) — идентификатор исполнителя
- `assigned_by` (UUID) — кто назначил (идентификатор пользователя)
- `assigned_at` (DateTime) — время назначения

**Выходные данные:**
- `task_id` (UUID) — идентификатор задачи
- `assignee_id` (UUID) — идентификатор назначенного исполнителя
- `assignee_email` (String) — email исполнителя
- `assignee_role` (Enum) — роль исполнителя в проекте
- `assigned_by` (UUID) — кто назначил
- `assigned_at` (DateTime) — время назначения
- `notification_sent` (Boolean) — отправлено ли уведомление

**Бизнес-логика:**
1. Назначать исполнителя можно только из участников проекта.
2. Только пользователи с ролями `Владелец` или `Редактор` могут назначать исполнителей.
3. Исполнитель получает уведомление о назначении на задачу.
4. Одна задача может иметь только одного исполнителя.
5. Исполнитель может быть переназначен (история назначений сохраняется).
6. Если исполнитель покидает проект, задача остаётся без исполнителя.


- **Этап 3:** Валидация прав доступа (только участники проекта)

### 6.3. Фича: Ролевая модель доступа

**Для чего нужна эта фича:** Разграничение прав доступа участников проекта для обеспечения безопасности и контроля над данными.

**Какую проблему она решает:** Предотвращает несанкционированные действия с задачами и проектами, обеспечивает принцип минимальных привилегий (пользователи имеют только необходимые права), позволяет гибко управлять доступом в зависимости от роли участника в проекте.

**Дополнительное описание:** Система определяет три роли: Владелец (полный доступ), Редактор (создание/изменение задач), Наблюдатель (только просмотр). Права проверяются при каждом запросе к API. Владелец может изменять роли участников и передавать владение.

**Входные данные:**
- `project_id` (UUID) — идентификатор проекта
- `user_id` (UUID) — идентификатор пользователя
- `current_role` (Enum) — текущая роль пользователя
- `requested_action` (String) — запрашиваемое действие (view, create, edit, delete, manage_members)
- `target_resource` (String) — целевой ресурс (task, project, member)

**Выходные данные:**
- `access_granted` (Boolean) — предоставлен ли доступ
- `user_role` (Enum) — роль пользователя
- `allowed_actions` (Array) — список разрешённых действий
- `restricted_actions` (Array) — список запрещённых действий

**Бизнес-логика:**
1. **Владелец:** полный доступ ко всем ресурсам проекта, включая управление участниками и удаление проекта.
2. **Редактор:** создание, просмотр, редактирование и удаление задач; просмотр участников; не может управлять участниками или удалить проект.
3. **Наблюдатель:** только просмотр задач и участников; не может создавать, редактировать или удалять задачи.
4. Проверка прав выполняется для каждого запроса к API.
5. При изменении роли права обновляются немедленно.
6. Владелец не может быть понижен в роли, пока не передаст владение другому.

## 8. Умная система напоминаний и уведомлений

### 8.1. Фича: Создание персонализированных напоминаний

**Входные данные:**
- `task_id` (UUID) — идентификатор задачи
- `user_id` (UUID) — идентификатор пользователя
- `reminder_type` (Enum) — тип напоминания: `static`, `relative`, `smart`, `escalation`
- `trigger_time` (DateTime) — конкретное время срабатывания (для static)
- `offset_minutes` (Integer) — смещение в минутах от дедлайна (для relative)
- `channels` (Array) — каналы доставки: `email`, `push`, `sms`
- `is_recurring` (Boolean) — является ли повторяющимся
- `recurrence_pattern` (Enum) — паттерн повторения: `daily`, `weekly`, `monthly`
- `snooze_enabled` (Boolean) — разрешена ли функция snooze
- `max_attempts` (Integer) — максимальное количество попыток

**Выходные данные:**
- `reminder_id` (UUID) — идентификатор созданного напоминания
- `task_id` (UUID) — идентификатор задачи
- `reminder_type` (Enum) — тип напоминания
- `next_trigger` (DateTime) — время следующего срабатывания
- `status` (Enum) — статус: `active`, `snoozed`, `completed`, `cancelled`
- `channels` (Array) — выбранные каналы доставки
- `created_at` (DateTime) — дата создания

**Бизнес-логика:**
1. Напоминание может быть создано только для существующей задачи.
2. Для типа `relative` время срабатывания вычисляется автоматически на основе дедлайна задачи.
3. Для типа `smart` система предлагает оптимальное время на основе истории пользователя.
4. Повторяющиеся напоминания создаются с указанным интервалом до выполнения задачи.
5. Максимальное количество попыток ограничивает число отправлений для одной задачи.
6. Пользователь не может создать напоминание в прошлом.

**Примеры JSON:**

*Входные данные (POST /api/reminders)*
```json
{
  "task_id": "task_abc123xyz789",
  "user_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "reminder_type": "relative",
  "offset_minutes": 60,
  "channels": ["email", "push"],
  "is_recurring": false,
  "snooze_enabled": true,
  "max_attempts": 3
}
```

*Выходные данные (ответ на создание напоминания)*
```json
{
  "reminder_id": "rem_xyz789abc123",
  "task_id": "task_abc123xyz789",
  "reminder_type": "relative",
  "next_trigger": "2024-01-15T08:00:00Z",
  "status": "active",
  "channels": ["email", "push"],
  "created_at": "2024-01-14T10:00:00Z"
}
```

**Поэтапная разработка:**
- **Этап 1:** CRUD операции для напоминаний
- **Этап 2:** Поддержка типов static и relative
- **Этап 3:** Реализация recurring напоминаний
- **Этап 4:** Интеграция с каналами доставки

### 8.2. Фича: Автоматические напоминания о дедлайнах

**Входные данные:**
- `task_id` (UUID) — идентификатор задачи
- `due_date` (DateTime) — дедлайн задачи
- `priority` (Enum) — приоритет задачи: `Низкий`, `Средний`, `Высокий`
- `current_time` (DateTime) — текущее время
- `user_notification_settings` (JSON) — настройки пользователя

**Выходные данные:**
- `reminder_id` (UUID) — идентификатор созданного напоминания
- `trigger_time` (DateTime) — рассчитанное время срабатывания
- `priority_multiplier` (Decimal) — коэффициент приоритета
- `auto_generated` (Boolean) — флаг автоматической генерации

**Бизнес-логика:**
1. Система автоматически создаёт напоминания для задач с дедлайном.
2. Время напоминания рассчитывается на основе приоритета:
    - Высокий: за 24 часа и за 2 часа до дедлайна
    - Средний: за 12 часов до дедлайна
    - Низкий: за 1 час до дедлайна
3. Если у пользователя включены тихие часы, напоминание переносится на начало активного периода.
4. Автоматические напоминания создаются только если пользователь не создал свои.

**Примеры JSON:**

*Входные данные (внутреннее событие при создании/обновлении задачи)*
```json
{
  "task_id": "task_abc123xyz789",
  "due_date": "2024-01-15T10:00:00Z",
  "priority": "Высокий",
  "current_time": "2024-01-14T10:00:00Z",
  "user_notification_settings": {
    "quiet_hours": {"start": "22:00", "end": "08:00"},
    "preferred_channels": ["push"]
  }
}
```

*Выходные данные (автоматически созданное напоминание)*
```json
{
  "reminder_id": "rem_auto123456789",
  "task_id": "task_abc123xyz789",
  "trigger_time": "2024-01-14T10:00:00Z",
  "priority_multiplier": 2.0,
  "auto_generated": true,
  "channels": ["push"]
}
```

**Поэтапная разработка:**
- **Этап 1:** Триггер на создание/обновление задачи с дедлайном
- **Этап 2:** Расчёт времени напоминания по приоритету
- **Этап 3:** Учёт тихих часов при планировании
- **Этап 4:** Проверка на существование пользовательских напоминаний

### 8.3. Фича: Мультиканальная доставка уведомлений

**Входные данные:**
- `reminder_id` (UUID) — идентификатор напоминания
- `channels` (Array) — целевые каналы: `email`, `push`, `sms`
- `message_content` (String) — текст уведомления
- `user_timezone` (String) — часовой пояс пользователя
- `delivery_priority` (Enum) — приоритет доставки: `normal`, `urgent`

**Выходные данные:**
- `delivery_results` (Array) — результаты по каждому каналу
    - `channel` (Enum) — название канала
    - `status` (Enum) — статус: `sent`, `delivered`, `failed`
    - `sent_at` (DateTime) — время отправки
    - `delivered_at` (DateTime) — время доставки (если применимо)
    - `error_message` (String) — сообщение об ошибке (если есть)

**Бизнес-логика:**
1. Отправка производится параллельно во все указанные каналы.
2. Каждый канал имеет своего провайдера и настройку retry-логики.
3. При неудачной отправке делается до 3 попыток с экспоненциальной задержкой.
4. Статус доставки отслеживается для каждого канала отдельно.
5. Канал SMS используется только для критических уведомлений (высокий приоритет).

**Примеры JSON:**

*Входные данные (внутренний запрос на отправку)*
```json
{
  "reminder_id": "rem_xyz789abc123",
  "channels": ["email", "push", "sms"],
  "message_content": "Напоминание: Подготовить презентацию (дедлайн через 1 час)",
  "user_timezone": "UTC+3",
  "delivery_priority": "urgent"
}
```

*Выходные данные (результаты доставки)*
```json
{
  "delivery_results": [
    {
      "channel": "email",
      "status": "delivered",
      "sent_at": "2024-01-15T08:00:00Z",
      "delivered_at": "2024-01-15T08:00:12Z",
      "error_message": null
    },
    {
      "channel": "push",
      "status": "delivered",
      "sent_at": "2024-01-15T08:00:01Z",
      "delivered_at": "2024-01-15T08:00:03Z",
      "error_message": null
    },
    {
      "channel": "sms",
      "status": "failed",
      "sent_at": "2024-01-15T08:00:02Z",
      "delivered_at": null,
      "error_message": "Недостаточно средств на счету SMS-провайдера"
    }
  ]
}
```

**Поэтапная разработка:**
- **Этап 1:** Интеграция с email-провайдером (SendGrid, SES, etc.)
- **Этап 2:** Интеграция с push-сервисом (Firebase, OneSignal)
- **Этап 3:** Интеграция с SMS-провайдером (Twilio, SMS.ru)
- **Этап 4:** Реализация retry-логики и мониторинга доставки

### 8.4. Фича: Умная snooze-функция

**Входные данные:**
- `reminder_id` (UUID) — идентификатор напоминания
- `user_id` (UUID) — идентификатор пользователя
- `snooze_action` (Enum) — действие: `snooze_5min`, `snooze_15min`, `snooze_1h`, `custom`
- `custom_snooze_until` (DateTime) — кастомное время (если выбрано custom)
- `previous_snooze_count` (Integer) — количество предыдущих откладываний

**Выходные данные:**
- `reminder_id` (UUID) — идентификатор напоминания
- `new_trigger_time` (DateTime) — новое время срабатывания
- `snooze_count` (Integer) — общее количество откладываний
- `snooze_limit_reached` (Boolean) — достигнут ли лимит
- `adaptive_interval_used` (Boolean) — использован ли адаптивный интервал

**Бизнес-логика:**
1. Пользователь может отложить напоминание на стандартные интервалы (5, 15, 30 минут, 1 час).
2. Адаптивная система предлагает интервал на основе истории поведения пользователя.
3. Максимальное количество откладываний по умолчанию равно 3 (настраивается).
4. После достижения лимита напоминание становится постоянным до выполнения или отмены.
5. Интервал может увеличиваться при каждом последующем snooze.

**Примеры JSON:**

*Входные данные (POST /api/reminders/{reminder_id}/snooze)*
```json
{
  "reminder_id": "rem_xyz789abc123",
  "user_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "snooze_action": "snooze_15min",
  "previous_snooze_count": 1
}
```

*Выходные данные (ответ на snooze)*
```json
{
  "reminder_id": "rem_xyz789abc123",
  "new_trigger_time": "2024-01-15T08:30:00Z",
  "snooze_count": 2,
  "snooze_limit_reached": false,
  "adaptive_interval_used": false
}
```

**Поэтапная разработка:**
- **Этап 1:** Базовая snooze-функция с фиксированными интервалами
- **Этап 2:** Ограничение максимального количества откладываний
- **Этап 3:** Сбор данных для адаптивного обучения
- **Этап 4:** Рекомендации адаптивных интервалов на основе ML

### 8.5. Фича: Групповые уведомления для командных проектов

**Входные данные:**
- `project_id` (UUID) — идентификатор проекта
- `task_id` (UUID) — идентификатор просроченной задачи
- `assignee_id` (UUID) — идентификатор исполнителя
- `overdue_hours` (Integer) — количество часов просрочки
- `escalation_level` (Enum) — уровень эскалации: `level_1`, `level_2`, `level_3`

**Выходные данные:**
- `notification_id` (UUID) — идентификатор группового уведомления
- `recipients` (Array) — список получателей
    - `user_id` (UUID) — идентификатор пользователя
    - `role` (Enum) — роль в проекте
    - `notification_channel` (Enum) — канал доставки
- `escalation_action` (String) — действие эскалации
- `message` (String) — текст уведомления

**Бизнес-логика:**
1. При просрочке задачи исполнитель получает первое уведомление.
2. Через 24 часа просрочки уведомление отправляется владельцу проекта (эскалация уровня 1).
3. Через 48 часов уведомляются все участники проекта с ролью Редактор и выше (уровень 2).
4. Через 72 часа проект помечается как «проблемный», уведомляется владелец компании (уровень 3).
5. Эскалация останавливается при выполнении задачи.

**Примеры JSON:**

*Входные данные (внутреннее событие эскалации)*
```json
{
  "project_id": "proj_abc123xyz789",
  "task_id": "task_abc123xyz789",
  "assignee_id": "user_def456uvw012",
  "overdue_hours": 48,
  "escalation_level": "level_2"
}
```

*Выходные данные (групповое уведомление)*
```json
{
  "notification_id": "notif_group123456",
  "recipients": [
    {
      "user_id": "user_def456uvw012",
      "role": "Редактор",
      "notification_channel": "push"
    },
    {
      "user_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
      "role": "Владелец",
      "notification_channel": "email"
    }
  ],
  "escalation_action": "notify_all_editors",
  "message": "Задача 'Подготовить презентацию' просрочена на 48 часов. Требуется внимание команды."
}
```

**Поэтапная разработка:**
- **Этап 1:** Детектирование просроченных задач в проектах
- **Этап 2:** Уведомление исполнителя о просрочке
- **Этап 3:** Реализация многоуровневой эскалации
- **Этап 4:** Групповая рассылка участникам проекта

*   **Иерархия:** `Задачи` могут принадлежать `Проекту` или быть личными (при `project_id = NULL`).
